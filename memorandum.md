# Padea Catering Operations Memo

My first step was not to write code. I reread the scenario several times to understand the shape of the problem: who the actors were, where information flowed, and where the process could break. I paid special attention to words like “may”, “usually”, “sometimes”, and “rarely”, because those words often hide the edge cases that break real operations systems. The process involved families, students, managers, caterers, coordinators, school exclusions, absences, preferences, dietary constraints, delivery details, and confirmation failures. My conclusion was that the stated problem was not simply “send weekly food orders.” The deeper problem was that too much operational memory and judgement was concentrated in the coordinator.

I also tried to apply Elon Musk’s “delete first” principle. I asked: why does catering exist at all? Removing it would probably not collapse the business overnight. But I decided it was still worth preserving. Catering reduces pressure on parents, differentiates Padea from other tutoring providers, gives students energy and a social break after a long day, and acts as a small reward for showing up and working hard. However, those benefits only exist if students get food they can actually eat and enjoy. If a student has an allergy, or receives a meal they hate, the “reward” becomes a liability. This led to one of my core requirements: the system should improve the chance that each student receives a suitable meal, rather than relying on the coordinator’s “educated guess.”

The biggest early insight came from the task sheet’s description of the coordinator manually emailing caterers, choosing a few menu items, and guessing quantities. This creates two bottlenecks. First, the coordinator’s work scales with the number of sessions. Second, student preference knowledge either lives in someone’s memory, scattered feedback notes, or not at all. My first design decision was therefore simple: collect each student’s food preference during onboarding. Dylan confirmed that onboarding already happens through a form, so adding food preference and dietary requirement fields is a low-friction change. This alone dramatically improves the process: instead of guessing, the system can query known preferences.

I deliberately tried to minimise effort for students, tutors, and managers. The system should be mostly invisible. Students should not be forced to fill out constant forms, and managers should not need to know how to handle every operational edge case. For that reason, I chose to collect only the primary caterer preference upfront. Collecting preferences for every possible backup caterer would create unnecessary friction. If a backup caterer is ever required, the system can then reach out with a one-off SMS/email form. This pushes friction into the rare case rather than the normal case.

The implementation became a Supabase-backed prototype. I modelled sessions, students, student-session membership, caterers, menus, food profiles, absences, exclusions, managers, caterer contacts, orders, order lines, caterer-school capacity, and exceptions. I used Python for deterministic order-generation logic and Streamlit for a lightweight coordinator dashboard. The weekly generator reads database state, applies absences and school exclusions, resolves attending students, selects preferred or backup meals, checks dietary compatibility, aggregates order quantities, creates email drafts for caterers, and produces distribution sheets for managers.

A key design choice was to keep the core ordering logic deterministic and auditable. I did not want an LLM deciding what a student with an allergy should eat. Instead, LLMs belong at the messy boundary: parsing family emails, manager notes, or ambiguous preference updates into structured proposed changes. The database remains the source of truth, and safety-critical changes can still be reviewed by a human.

A major learning was that complexity hides in the “last 1%.” The initial form-based solution handles most cases, but edge cases quickly explode: absent students, mid-semester changes, full school cancellations, partial year-level exclusions, missing manager contacts, missing caterer contacts, caterer minimum quantities, backup caterers, and confirmation failures. I built an exception system so the software does not silently pretend everything is fine. For example, missing mobile numbers, unsafe meal assignments, below-minimum orders, and overdue caterer confirmations become visible exceptions in the dashboard.

I also modelled the order lifecycle. An order moves from ready for review, to approved, to sent to caterer, to confirmed or overdue. If the caterer fails to confirm by the deadline, the system can mark the order overdue and escalate toward a backup caterer. I separated “confirmation overdue” from “backup required” because the first is a factual state, while the second is an operational decision. A late caterer reply might still be accepted.

Not everything went smoothly. I repeatedly ran into schema decisions that seemed elegant in theory but annoying in practice. Dropping duplicated fields like `orders.school_id` improved normalisation but forced code refactoring. I also questioned whether some tables were worth storing at all. The useful distinction I arrived at was: duplicated truth is usually bad, but decision snapshots are valuable. For example, `orders.school_id` is recoverable from the session, but a future `order_student_map` may be worth storing because it records which student was assigned which meal at the time the order was generated.

The current prototype is not production-ready. It does not send real emails or SMS messages, and family email handling is simulated. Preference updates and absence updates currently demonstrate the state change by regenerating orders, whereas production should regenerate only the affected draft order or create an exception if the order has already been sent. Still, the prototype demonstrates the core operating loop: database state in, draft orders and exceptions out, coordinator reviews, caterer confirmation is tracked, and family updates change future orders.

The main lesson was that operations engineering is less about writing a clever script and more about making hidden state explicit. Once preferences, absences, exclusions, contacts, menus, and confirmation status are modelled clearly, much of the coordinator’s repetitive work can be automated. The human role shifts from remembering and guessing to reviewing exceptions and making judgement calls.

I left with plenty of extension ideas, but I deliberately stopped short of chasing every edge case. The important deliverable was not a perfect catering SaaS product; it was a working operations prototype that made the core state explicit. In production, I would add scoped regeneration of only the affected order after an absence/preference update, real email ingestion from families@padea.com.au, SMS or form-based backup preference collection, a proper coordinator dashboard with action buttons, an audit log of order state transitions, stronger structured allergy modelling, persistent student-meal assignment records for traceability, and many other ideas. The prototype proves the operating loop; these extensions would harden it.
